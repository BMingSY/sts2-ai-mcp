# Compatibility Policy

Slay the Spire 2 updates can break mod binaries, game-data assumptions, and PCK compatibility. This repository treats compatibility as release metadata, not as an implicit promise.

## Release Metadata

Every public release should record:

- `mod_version`
- `v2_protocol_version`
- `state_version` and `decision_version`
- semantic capability flags such as `exact_character_ascension`, `preview_action`, `action_trace`, `monster_state_machine_export`, `unified_trigger_progress`, `run_analysis`, `map_boss_identity`, and `selection_runtime_preview`
- tested Slay the Spire 2 game version
- tested Godot/MegaDot version
- platform and architecture
- build date

Recommended release artifact name:

```text
sts2-ai-mcp-v0.1.0-sts2-<game-version>-godot-4.5.1.zip
```

## Runtime Health

The mod health response should expose enough data for MCP clients to decide whether normal play is safe:

```json
{
  "service": "sts2-ai-mcp",
  "mod_version": "0.1.7",
  "v2_protocol_version": "2026-07-18-v2-draft",
  "state_version": 13,
  "decision_version": 6,
  "game_version": "v0.107.1",
  "capabilities": {
    "run_analysis": true,
    "map_boss_identity": true,
    "selection_runtime_preview": true
  },
  "compatibility": {
    "status": "untested",
    "supported_game_versions": [],
    "notes": []
  }
}
```

Status meanings:

- `compatible`: tested against this exact game build or an explicitly supported version range.
- `untested`: game version is readable, but this mod build has not been validated against it.
- `incompatible`: known incompatible game build or engine/runtime mismatch.

AI-safe clients should refuse normal `take_action` calls when status is `incompatible`, and should surface a warning when status is `untested`.

The bundled MCP now performs a strict startup contract check for protocol `2026-07-18-v2-draft`, state version 13, decision version 6, and its required capability flags. It exits instead of serving an older contract silently. An explicit `STS2_MCP_ALLOW_INCOMPATIBLE=1` override is available only for local compatibility experiments.

## Game Data

Live game model data is preferred over checked-in cache data.

If cache data is used, responses should include:

- `data_source`
- `game_version`
- `content_hash`
- stale/unknown status when the running game version does not match the cache

Private or generated gameplay knowledge is not repository content. Keep it outside git or under ignored paths such as `agent_knowledge/` and `private_knowledge/`.

## Godot/MegaDot

Current target: Godot/MegaDot `4.5.1.m.12`.

Do not use Godot `4.6.x` to pack the mod PCK. The game may reject PCK files generated by newer Godot versions.

Build scripts should:

- prefer the game-bundled executable/runtime when possible
- reject obvious `4.6.x` packers by default
- print the detected Godot/MegaDot version
- require an explicit override before allowing a version mismatch
